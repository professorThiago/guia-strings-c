Introdução

Por que strings em C são diferentes

Em linguagens como Python ou JavaScript, texto é um tipo nativo: você cria, concatena e descarta strings sem pensar em memória. Em C não existe um tipo string embutido. Texto é representado por uma sequência de bytes na memória, e cabe a você gerenciar onde esses bytes vivem e por quanto tempo.

Isso assusta no começo, mas é exatamente o que dá a C seu controle preciso — essencial em sistemas embarcados, onde cada byte de RAM conta. Este guia mostra os dois caminhos: o jeito C cru (char* e arrays) e a conveniência da classe String do Arduino, deixando claro quando usar cada um.

Como ler este guia

Cada seção tem um exemplo compilável. Se você está no ESP32/Arduino, pode colar os trechos direto no setup() e ver a saída no Serial Monitor. Os conceitos de char* valem para C puro também.

Seção 01

O que é uma string em C

Uma string em C é simplesmente um array de caracteres terminado por um byte nulo — o caractere '\0' (valor zero). Esse terminador é como as funções sabem onde o texto acaba.

A palavra "Oi" não ocupa 2 bytes na memória, e sim 3:

Memória da string "Oi"
0x00'O'
0x01'i'
0x02\0

Sem o '\0', funções como printf ou strlen continuariam lendo a memória além do texto, produzindo lixo ou travando o programa.

string_basica.cC
#include <stdio.h>
#include <string.h>

int main(void) {
    // As duas formas abaixo são equivalentes:
    char a[] = "Oi";              // compilador adiciona '\0'
    char b[] = {'O', 'i', '\0'};  // explícito

    printf("%s tem %zu caracteres\n", a, strlen(a));
    // Saída: Oi tem 2 caracteres  (strlen NÃO conta o '\0')

    printf("mas ocupa %zu bytes\n", sizeof(a));
    // Saída: mas ocupa 3 bytes  (sizeof conta o '\0')

    return 0;
}
Diferença crucial

strlen() conta os caracteres até o '\0' (não o inclui). sizeof() conta o espaço total reservado, incluindo o terminador. Confundir os dois é fonte clássica de bugs.

Seção 02

Array de char: a string mutável

Quando você declara char buffer[N], o compilador reserva N bytes contíguos, geralmente na stack (pilha). Esse espaço é seu para ler e escrever — é a forma preferida para texto que muda.

array_char.cC
char nome[20];           // reserva 20 bytes, conteúdo indefinido

strcpy(nome, "Thiago");  // copia "Thiago\0" para dentro
nome[0] = 't';           // pode modificar caractere a caractere

printf("%s\n", nome);   // thiago

// Inicialização direta — tamanho deduzido automaticamente
char saudacao[] = "Bom dia";  // 8 bytes (7 letras + '\0')

Por que dimensionar com folga

O array tem tamanho fixo definido em tempo de compilação. Se você precisa montar uma string de até 15 caracteres, reserve pelo menos 16 bytes — os 15 visíveis mais o '\0'. Escrever além do limite é buffer overflow, um dos bugs mais perigosos em C.

Regra de ouro

Para guardar N caracteres visíveis, declare o array com N+1 bytes. Sempre reserve espaço para o terminador.

Seção 03

O ponteiro char*

Um char* não armazena texto — ele armazena o endereço do primeiro caractere de uma string em algum lugar da memória. Entender essa distinção é o ponto que separa quem entende C de quem apenas copia exemplos.

Os dois usos de char*

1. Apontar para um literal (somente leitura)

ptr_literal.cC
const char *msg = "Imutável";
// 'msg' aponta para um literal armazenado em memória somente-leitura.
// Tentar modificar é comportamento indefinido:
// msg[0] = 'x';   ← NÃO FAÇA — pode travar (crash)

printf("%s\n", msg);  // ler é seguro
Armadilha clássica

Escrever char *msg = "texto"; e depois tentar msg[0] = 'X' compila, mas trava em tempo de execução. Literais ficam em memória protegida. Use sempre const char* para deixar essa intenção clara — o compilador então avisa se você tentar modificar.

2. Apontar para um array que você controla

ptr_array.cC
char buffer[32];
char *p = buffer;     // p aponta para o início do buffer

strcpy(p, "editável");  // agora pode modificar via ponteiro
p[0] = 'E';            // OK — buffer é seu

// Aritmética de ponteiro: avançar pela string
while (*p != '\0') {
    putchar(*p);     // imprime caractere atual
    p++;              // avança para o próximo
}

Seção 04

As funções da string.h

A biblioteca padrão <string.h> traz as ferramentas essenciais. Todas dependem do terminador '\0' para saber onde parar.

FunçãoO que fazCuidado
strlen(s)Conta caracteres até o '\0'Não inclui o terminador
strcpy(d,s)Copia s para dNão checa tamanho de d
strncpy(d,s,n)Copia no máximo n bytesPode não terminar com '\0'
strcat(d,s)Concatena s ao fim de dd precisa ter espaço
strcmp(a,b)Compara; 0 se iguaisNão é booleano!

funcoes_string.cC
#include <string.h>
#include <stdio.h>

int main(void) {
    char destino[50] = "Olá, ";
    const char *nome = "mundo";

    strcat(destino, nome);        // "Olá, mundo"
    printf("%s\n", destino);

    // Comparação — ATENÇÃO: 0 significa IGUAIS
    if (strcmp(nome, "mundo") == 0) {
        printf("São iguais!\n");
    }

    // Buscar um caractere
    char *virgula = strchr(destino, ',');
    if (virgula) {
        printf("Vírgula na posição %ld\n", virgula - destino);
    }
    return 0;
}
strcmp não é booleano

Iniciantes escrevem if (strcmp(a, b)) esperando "se iguais". É o oposto: strcmp retorna 0 quando são iguais, e if (0) é falso. O correto é if (strcmp(a, b) == 0).

Seção 05

Formatação segura com snprintf

Para montar strings com números, variáveis e formatação, snprintf é a ferramenta certa. Diferente de sprintf, ele recebe o tamanho do buffer e nunca escreve além dele — eliminando overflow.

snprintf.cC
char buf[32];
int   temp = 22;
float umid = 65.4;

// snprintf(destino, tamanho, formato, ...args)
snprintf(buf, sizeof(buf), "%d°C / %.1f%%", temp, umid);
// buf agora contém: "22°C / 65.4%"

// Montar um tópico MQTT dinâmico
char topico[64];
int sala = 3;
snprintf(topico, sizeof(topico), "classroom/sala-%02d/status", sala);
// "classroom/sala-03/status"

Por que snprintf no embarcado

Em ESP32/Arduino, snprintf com buffer fixo é a base de bibliotecas eficientes: zero alocação dinâmica, tamanho previsível, sem fragmentação de heap. É exatamente o que substitui a classe String em código de produção.

Seção 06

Erros comuns e como evitá-los

1. Buffer pequeno demais

erro_overflow.cC
char nome[5];
strcpy(nome, "Thiago");  // ✗ "Thiago\0" = 7 bytes em 5 → OVERFLOW

// ✓ Correto: buffer com folga + função limitada
char nome2[16];
snprintf(nome2, sizeof(nome2), "%s", "Thiago");

2. Retornar ponteiro para variável local

erro_dangling.cC
char* criarNome(void) {
    char local[20] = "temporário";
    return local;   // ✗ 'local' deixa de existir ao sair da função
}                    // ponteiro retornado aponta para lixo

// ✓ Correto: receber buffer de quem chama
void criarNome2(char *destino, size_t n) {
    snprintf(destino, n, "%s", "permanente");
}

3. Esquecer o terminador

erro_terminador.cC
char buf[3] = {'O', 'i'};  // só 2 chars, sem '\0' explícito
// Neste caso o 3º byte é 0 por inicialização parcial — sorte.
// Mas em buffers não inicializados, o lixo continua sendo lido.

char dst[4];
strncpy(dst, "teste", 4);  // ✗ copia "test" SEM '\0'
dst[3] = '\0';            // ✓ garante terminação manual
Resumo dos perigos

Os três bugs acima — overflow, ponteiro pendente (dangling) e falta de terminador — respondem pela maioria dos crashes em código C com strings. A defesa é sempre a mesma: buffer com folga, funções com limite de tamanho, e terminação garantida.

Seção 07

A classe String do Arduino

No ecossistema Arduino/ESP32 existe a classe String (com S maiúsculo). Ela embrulha toda a complexidade de char* numa interface confortável — ao custo de alocar memória dinamicamente.

string_arduino.inoC++
String nome = "Thiago";
String saudacao = "Olá, " + nome;  // concatenação natural

saudacao += "!";                     // anexar
int tam = saudacao.length();        // tamanho

if (nome == "Thiago") {              // comparação com ==  (intuitivo!)
    Serial.println(saudacao);
}

// Conversões prontas
int n = String("42").toInt();        // texto → número
String s = String(3.14, 2);          // número → "3.14"

// Métodos úteis
saudacao.toUpperCase();              // maiúsculas
saudacao.indexOf('T');              // posição de um char
saudacao.substring(0, 3);           // recorte
Quando String brilha

Para prototipagem rápida, parsing de entradas curtas, ou código que roda poucas vezes (ex: dentro de setup()), String é prática e segura — você não corre risco de overflow. O problema aparece em uso repetido e prolongado, como veremos a seguir.

Seção 08

Regra prática

Use String em prototipagem e código que roda poucas vezes. Use char* + snprintf em loops, callbacks frequentes, drivers e bibliotecas — qualquer coisa que executa milhares de vezes ao longo de horas.

Seção 09

Fragmentação de heap: o perigo silencioso

Toda vez que uma String cresce, encolhe ou é criada, ela aloca e libera blocos no heap. Em um PC isso é trivial. Em um microcontrolador com poucos KB de RAM, alocações e liberações repetidas deixam buracos na memória.

Há 4 bytes livres (□) somados, mas espalhados. Se o programa pedir um bloco contíguo de 4 bytes, a alocação falha mesmo havendo espaço total — porque não há 4 bytes seguidos. Com o tempo, o dispositivo trava ou reinicia sozinho.

fragmentacao.inoC++
// ✗ PROBLEMÁTICO — roda no loop(), aloca a cada iteração
void loop() {
    String msg = "Temp: ";
    msg += lerTemperatura();
    msg += "°C";
    Serial.println(msg);   // nova alocação a cada volta → fragmenta
    delay(100);
}

// ✓ EFICIENTE — buffer fixo, zero alocação
void loop() {
    char msg[32];
    snprintf(msg, sizeof(msg), "Temp: %d°C", lerTemperatura());
    Serial.println(msg);   // mesma memória sempre → estável
    delay(100);
}
Sintoma real

Um projeto que funciona por minutos e depois "trava do nada" após horas rodando é o sintoma clássico de fragmentação de heap por uso intenso de String. A correção é migrar os caminhos quentes para char* + snprintf.

Seção 10

Boas práticas consolidadas

Padrão recomendado para funções

boas_praticas.cC
// Quem chama fornece o buffer e seu tamanho.
// A função nunca aloca nem retorna ponteiro interno.
void formatarStatus(char *dst, size_t n,
                    int id, bool ligado) {
    snprintf(dst, n, "modulo %d: %s",
             id, ligado ? "ON" : "OFF");
}

void exemplo(void) {
    char linha[40];
    formatarStatus(linha, sizeof(linha), 3, true);
    // linha = "modulo 3: ON"
}

Checklist mental ao lidar com strings

PerguntaSe sim →
Vai modificar o texto?Array de char, não literal
Roda muitas vezes?char* + snprintf, evite String

O equilíbrio

Use String onde a conveniência vale mais que a performance (setup, parsing pontual). Use char* onde a estabilidade importa (loops, ISRs, bibliotecas). Saber alternar entre os dois é o que define o programador C/embarcado maduro.

Seção 11

Conclusão

Strings em C parecem hostis no início porque expõem o que outras linguagens escondem: texto é memória, e memória precisa ser gerenciada. Mas é justamente esse controle que torna C insubstituível em sistemas embarcados.

Depois de internalizar três ideias — o terminador '\0', a diferença entre array e ponteiro, e o custo da memória dinâmica — você passa a escrever código que não só funciona, mas que continua funcionando depois de dias ligado. E essa é a verdadeira marca de software embarcado bem feito.

Próximo passo

Pegue um projeto seu que usa String dentro do loop() e migre para char[] + snprintf. Meça a RAM livre antes e depois com ESP.getFreeHeap() ao longo do tempo — a diferença na estabilidade é visível.
